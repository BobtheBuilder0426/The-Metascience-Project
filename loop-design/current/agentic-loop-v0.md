# Agentic Loop — v0 (flat single-session)  [CS] — 2026-07-09  (LB-021)

**What it is.** A blank Claude Code (CC), given only this loop, drives **one** Claude Science (CS) session through a
research question and returns a better answer than raw CS would produce on the same word-identical question. v0 is the
**simplest loop that is still a real loop** — plan → act → review → revise → integrate — with the four cheap,
always-on operator findings built in (B cadence, D plan-review, F keep-what-is-good, G literature+citations). The
expensive structural ideas (Mission-Control multi-session, the 2-lens×2-brain panel) are **deliberately excluded from
v0** so their benefit can be *measured* later (D2), not assumed.

## 0. Roles + the drive channel (unchanged from D1)
- **CC (the driver, blank):** runs this loop. Never touches the shared repo/labbook (isolation, D1 §2). Controls the CS
  browser tab (Chrome control). Holds the loop state + budget.
- **CS (the scientist, driven):** does the actual science — reasoning + the full featured-connector suite (PubMed,
  OpenAlex, Ensembl, Open Targets, UniProt, …) + skills. Answers in its own tab.
- **Human/the assistant:** only for the graceful-degradation approval point (D1 §6a) if autonomy can't be guaranteed.

## 0. The fairness model (operator-locked, LB-027) — what enters the loop
Both arms receive the **exact same start prompt: a naive research question, typed plainly by a curious scientist who
knows nothing about CS or this loop** (byte-identical; plus identical attached material where a question provides it).
- **Arm B (baseline):** naive question → blank CS → answer. No loop.
- **Arm L (loop):** the *same* naive question → blank CC (bootstrapped with a `CLAUDE.md` start package) → the CC
  **reframes the question itself** into a rigorous brief (STEP 0 below) and drives CS through the loop.

The loop's enrichment (finding G — "use your connectors, cite everything, show reasoning") is therefore **generated by
the CC from the naive question, not authored by a human**. The baseline never receives it. This makes the measured Δ
attributable to the loop machinery, not to a better-worded prompt — GOAL D4's "blank-CS vs blank-CC+loop, same
word-identical question," made rigorous. The CC saves its reframe as `<QID>_reframed_brief.md` (audit trail: only the
*approach* was enriched, never the *question asked*).

## 1. The loop (v0 — STEP 0 reframe + five phases, one CS session)

```
        ┌────────────────────────────────────────────────────────────────┐
        │  CC driver (blank) — holds plan, budget, KEEP-list, reframe      │
        └───────────────┬────────────────────────────────────────────────┘
   (0) REFRAME          │  CC (NOT CS): rewrite the SAME naive question into a rigorous research
                        │  brief — exploit full connector suite, require citations + reasoning chain,
                        │  state what a landmark answer needs. Does NOT change WHAT is asked. This is
                        │  where the loop earns its edge; the baseline gets no such reframe.
   (1) FRAME            │  CC → CS: send the reframed brief (+ any attached materials).
                        │           CS restates the question + success criteria. No answer yet.
   (2) PLAN             ▼
                CS drafts a PLAN (approach, which connectors/data, what a strong answer needs).
                        │
   (3) PLAN-REVIEW  ◄───┘  ← operator finding D (CHEAP, high-leverage; before any expensive work)
                CC asks CS to critique its OWN plan on 3 axes: (a) ambition — "what would make this a
                landmark answer, not a safe one?"; (b) grounding — "which claims will need a citation +
                a connector lookup?"; (c) risk — "where could this hallucinate?". CS revises the plan.
                        │
   (4) ACT             ▼
                CS executes the revised plan: retrieves literature (PubMed/OpenAlex), pulls data via
                connectors, reasons, and DRAFTS the answer — every non-trivial claim carries a citation
                (finding G). CS shows its chain of reasoning (feeds the creativity reasoning-trace gate).
                        │
   (5) RESULT-REVIEW ◄──┘  ← KEEP-FIRST (operator finding F), then correctness, then improve
                CC runs the review gate ON the draft:
                  5a. KEEP: CS lists what is strong + WHY, and marks it protected (do not regress).
                  5b. CORRECTNESS: CS self-checks every citation (exists? supports the claim?), every
                      dataset/entity named, every number — anti-hallucination pass (mirrors the harness).
                  5c. IMPROVE: only now, CS proposes evidence-backed improvements (never vibe edits),
                      applies the ones that don't regress the KEEP-list.
                        │
        INTEGRATE       ▼
                CS produces the FINAL answer + a short self-scorecard (the 5 rubric dimensions, its own
                citations list, and its reasoning trace for any novel claim). CC returns it.
```

**One revision cycle in v0** (phases 4–5 run once, with the plan-review at 3 being the cheap extra pass). More cycles =
a tunable we test later; v0 keeps it at one to bound `claude -p` budget (finding A/B) and to be a clean baseline.

## 2. The four always-on operator findings, and HOW v0 encodes each
| Finding | v0 encoding |
|---|---|
| **B — cadence + permissions** | CC polls the CS tab **every ≥5 min (default 10)**, not every minute; and does a **one-time project-wide permission pre-grant** at the start (code=Always, connectors=per-project, compute=all-jobs), so residual approval cards are rare. A watchdog check on each poll handles any straggler card. |
| **D — plan-stage review** | Phase 3 is a dedicated cheap review BEFORE the expensive ACT — inject ambition + flag hallucination risk when it costs almost nothing. |
| **F — keep what is good** | Phase 5a is mandatory and FIRST: name strengths + why, protect them; every later edit must not regress the KEEP-list. |
| **G — literature + citations** | Standing context brief tells CS to exploit the full connector suite; every non-trivial claim must carry a verifiable citation + show reasoning (this is exactly what the scoring harness rewards). |

## 3. What v0 deliberately EXCLUDES (so we can measure it later — D2)
- **Mission-Control multi-session (finding A/C)** → improvement category **C1**. v0 is one session; when context pressure
  shows up on a real run, C1 is the first thing we test against this baseline.
- **2-lens × 2-brain panel + Codex (finding E)** → improvement category **C3**. v0's review is CS reviewing its own
  work (single brain); adding the ambitious-postdoc/mentoring-PI panel across Claude+Codex is a measured upgrade.
- **CS Specialists / project-context depth** → **C4**; deeper connector chaining → **C5**; idea recombination → **C7**.

## 4. Success definition for v0 (what "the loop works" means at exp-001)
On the aging test-set, **Arm L (blank CC + this v0 loop → CS)** must beat **Arm B (raw blank CS, same word-identical
question)** on the weighted composite (rubric.json), scored by the harness, across ≥2 runs, with the gap larger than the
run-to-run spread. v0 does not need to win big — it needs to win **measurably and honestly**, establishing the baseline
that the improvement categories (C1–C7) then widen. That is exactly what exp-001 tests.

## 5. Honesty guards baked into v0
- The loop cannot claim a citation it didn't verify (phase 5b mirrors the harness's two-stage check).
- The loop cannot score its own creativity high without a legible reasoning trace (phase 4 shows the chain; the harness
  gates it).
- KEEP-first (5a) stops the loop from "improving" a good answer into a worse, over-engineered one (finding F).
- Everything CS produces is answerable on the word-identical question a raw CS baseline also gets — no privileged input.
