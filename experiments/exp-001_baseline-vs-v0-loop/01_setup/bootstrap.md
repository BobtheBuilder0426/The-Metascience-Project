# exp-001 — Bootstrap explainer: the CC start package + the fairness model  [CS-authored]

## The core fairness model (operator-locked, LB-027)
Both arms start from the **exact same start prompt: a naive research question, typed plainly by a curious scientist who
knows nothing about CS or this loop.** The two arms differ ONLY in what machinery sits behind that question:

```
                          the SAME naive question (byte-identical)  ── (+ same PDFs for Q_ERGO)
                                       │
                 ┌─────────────────────┴─────────────────────┐
                 ▼                                            ▼
   ARM B (baseline)                              ARM L (loop)
   naive question → blank CS → answer            naive question → blank CC  (has workspace/CLAUDE.md)
   (no loop, no reframe, one shot)                          │
                                                            ▼
                                                 CC REFRAMES the question into a rigorous
                                                 brief, then drives CS through the 5-phase
                                                 loop → final answer
```

The loop's advantage therefore comes **entirely from the CC's own work** — improving the question + running the loop —
never from a human handing the loop a better-worded prompt. That is what makes the measured Δ attributable to the
harness, not to prompt luck. This is GOAL D4's "blank-CS vs blank-CC+loop, same word-identical question," made rigorous.

## How the CC "knows" to do this — the CLAUDE.md mechanism
A blank Claude Code started in a folder **auto-reads `CLAUDE.md`** in that folder. So the CC start package IS the
`workspace/` folder, and `workspace/CLAUDE.md` is its operating manual. We hand a context-less CC this folder, start it
there, and give it the naive question — from `CLAUDE.md` alone it then knows to reframe the question and run the loop.
No other repo file is needed (isolation requirement).

**The CC start package (`workspace/`) contains:**
- `CLAUDE.md` — auto-loaded operating manual: identity, the fairness rule, ground rules, and the full runnable loop
  (STEP 0 REFRAME → FRAME → PLAN → PLAN-REVIEW → ACT → RESULT-REVIEW → INTEGRATE).
- `materials/` — the two Q_ERGO PDFs (attached to CS for every Q_ERGO run, both arms).
- `README.md` — the workspace file map (what the CC produces).
- (created by the CC at run time) `run_log.md`, `<QID>_reframed_brief.md`, the answer files, `RUN_REPORT.md`.

## Where the loop's "enrichment" now lives (the fix vs the earlier draft)
Earlier this bootstrap hand-wrote an enriched prompt ("use all your connectors, cite everything…") and sent it to CS.
That was wrong under the fairness model: it made the loop's input a *different, better* prompt authored by CS. Now that
enrichment is **generated by the CC** in STEP 0 REFRAME, from the naive question — see `workspace/CLAUDE.md`. The
baseline never sees it, because no CC reframes for the baseline. The `<QID>_reframed_brief.md` file the CC saves is the
audit trail proving it only enriched the *approach*, not the *question*.

## To actually run exp-001
The blank CC operates from `workspace/`; the run design (arms, #runs, file naming, what to attach) is in
`../protocol.md`; the naive questions are pinned in `../test-set/`. `../protocol.md` is the CS/the assistant-facing run
